================================================================================
CURRENT CODE FLOW: AFTER SHAP, ACTIONS, AND DOWNSTREAM AGENTIC PIPELINE
================================================================================

This document summarizes what the repository does after SHAP is computed in
Train.ipynb: mitigation summaries, inference JSON, RAG + LLM plans, and
scoring. It replaces older "Cell N" references with notebook names and file
paths that match the current tree.

================================================================================
PART 1: SHAP COMPUTATION AND GLOBAL ANALYSIS (Train.ipynb)
================================================================================

## Step 1: SHAP values
- Meta-model (on 192-dimensional concatenated agent embeddings) is explained with
  shap.KernelExplainer.
- Party-level attributions are aggregated from the three 64-dimensional blocks (default architecture).

## Step 2: Global / per-class analysis
- Mean |SHAP| per party globally and per attack class.
- Dominant party per class (highest mean contribution among the three agents).
- Timestamped exports, for example:
    outputs/vfl_shap_global_summary_<timestamp>.csv
    summary/vfl_shap_<class>_summary_<timestamp>.csv

(Exact filename patterns are built in the SHAP summary cells of Train.ipynb.)

Example-style console output (wording may vary):
  Volume_Rate_Sensor_Agent1 vs Packet_Size_Sensor_Agent2 vs
  Timing_Direction_Sensor_Agent3 contribution percentages per class.

================================================================================
PART 2: ACTION DETERMINATION (Train.ipynb — mitigation summary section)
================================================================================

## Step 1: Dominant agent per attack class
For each attack class (from explained test rows):
  - Mean SHAP magnitude (or contribution share) per party.
  - argmax -> dominant party index.

## Step 2: Primary action strings
ATTACK_ACTIONS is defined in utils/vfl_utils.py and imported in Train.ipynb.
It maps uppercase attack names to long mitigation strings (rate-limit, WAF,
etc.).

## Step 3: Per-party suggested actions
- Dominant party: primary line from ATTACK_ACTIONS for that attack type.
- Non-dominant parties: monitoring / secondary lines via agent_action_mapping
  built earlier with get_agent_actions_for_attack(...) over each agent feature
  list.

Code shape in Train.ipynb (paraphrased):
  suggested_action = ATTACK_ACTIONS.get(attack_type, ...)  # if dominant
  suggested_action = agent_action_mapping[i].get(attack_type, agent_actions[i])  # if not

( agent_action_mapping replaces older docs that said party_action_mapping. )

## Step 4: Human-readable formatting
format_action_readable() (and related helpers in utils/vfl_utils.py) can turn
comma-separated action strings into bullet-style text for reports.

## Step 5: Mitigation summary artifacts
Timestamped CSV / text-style outputs, e.g.:
  outputs/vfl_shap_agent_mitigation_summary_<timestamp>.csv
  outputs/vfl_shap_mitigation_summary.txt (when that cell writes it)

Columns typically include class, agent, mean contribution, suggested action,
is_dominant flag — see Train.ipynb for the exact DataFrame schema.

================================================================================
PART 4: WHAT THE TRAINING PIPELINE DOES NOT DO
================================================================================

Train.ipynb (and Predict.ipynb) do not execute network controls:
  - No firewall / WAF / rate-limit API calls.
  - No automated blocking or routing changes.
They produce analysis, model checkpoints, and structured recommendations only.

================================================================================
PART 5: INFERENCE AND AGENTIC RAG (Predict.ipynb -> Plan.ipynb -> Score.ipynb)
================================================================================

## Predict.ipynb
- Loads model/ checkpoints and scaler(s).
- Runs VFL forward pass + meta-model where applicable.
- Emits JSON under RAG_docs/predictions/ with per-sample labels, confidence,
  SHAP summaries, and tier/party context for downstream prompting.

## Plan.ipynb
- Loads prediction JSON + RAG_docs/knowledge/ (PDF/JSON).
- Chunks text (RecursiveCharacterTextSplitter, chunk_size 1000, overlap 200), embeds with
  sentence-transformers all-MiniLM-L6-v2, builds in-memory FAISS.
- build_llm_based_rag_query(...) uses OpenAI (OPENAI_MODEL, default gpt-4o-mini).
- search_rag_knowledge: top_k=10 similarity hits; top 5 chunks go to the LLM.
- Writes action_plan_*.json (and comparison files when enabled) under
  RAG_docs/action_plans/.

## Score.ipynb
- Loads action plans / comparison JSON.
- Reports ROUGE-1, BLEU, SBERT cosine, BERTScore-style metrics.

End-to-end order for experiments:
  Train.ipynb -> Predict.ipynb -> Plan.ipynb -> Score.ipynb
