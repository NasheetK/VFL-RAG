================================================================================
SHAP ANALYSIS: WHICH DATASET IS USED (TEST vs TRAIN)
================================================================================

This document explains which data your code uses for SHAP computation and
analysis. All behavior described here is implemented in Train.ipynb (VFL +
meta-model + SHAP sections).

================================================================================
ANSWER: SHAP IS COMPUTED ON TEST SAMPLES
================================================================================

The pipeline uses:
- BACKGROUND (for SHAP explainer): TRAINING meta-features (X_train_meta)
- SAMPLES TO EXPLAIN: TEST meta-features (X_test_meta)
- LABELS FOR ANALYSIS: TEST labels (y_test)

This is the standard approach: train-derived baseline, explanations on held-out
test rows.

================================================================================
DETAILED BREAKDOWN
================================================================================

## Step 1: Background dataset (for SHAP explainer)

In Train.ipynb, after X_train_meta is built, the notebook samples a random
subset of training rows for the KernelExplainer background:

  bg_size = min(100, X_train_meta.shape[0])
  bg_idx = torch.randperm(X_train_meta.shape[0])[:bg_size]
  background = X_train_meta[bg_idx].detach().cpu().numpy()

The same background array is also saved as model/shap_background.npy for
optional reuse (e.g. faster workflows in Predict.ipynb).

Why training data for background:
- SHAP needs a reference distribution; training meta-features match what the
  meta-model was fit on.
- This is common practice for KernelExplainer-style setups.

## Step 2: Samples to explain

Still in Train.ipynb:

  n_explain = min(200, X_test_meta.shape[0])
  X_explain = X_test_meta[:n_explain].detach().cpu().numpy()

So the explained rows are the first n_explain test meta-feature vectors
(default cap 200).

Why test data:
- Reflects generalization behavior on data not used for training the VFL
  encoders (test split is held out at the flow level).

## Step 3: Labels for grouping

Labels aligned to X_explain come from y_test (test split), e.g. y_explain =
y_test_np[:n_explain], so per-class SHAP summaries match the explained test
rows.

## Step 4: SHAP computation

  explainer = shap.KernelExplainer(meta_predict, background)
  shap_values = explainer.shap_values(X_explain, nsamples=200)

SHAP values are computed for party-level blocks derived from the 192-dimensional
concatenated embedding (3 agents x 64 dimensions each in the default config).

================================================================================
VISUAL FLOW DIAGRAM
================================================================================

Data split (stratified, from Train.ipynb):

  Full dataset (e.g. ~390k flows, 88 features)
 |
       +-- TRAIN (64%)
       +-- VALIDATION (16%)
       +-- TEST (20%)

Meta-model inputs:
  X_train_meta  = concat(agent train embeddings)  -> (N_train, 192)
  X_test_meta   = concat(agent test embeddings)   -> (N_test, 192)

SHAP (in Train.ipynb):
  Background: 100 rows sampled from X_train_meta
  Explain:    up to 200 rows from X_test_meta
  Labels:     y_test aligned to those rows

================================================================================
OUTPUT ARTIFACTS (TRAIN.IPYNB)
================================================================================

SHAP-related exports use timestamped filenames, for example:
 outputs/vfl_shap_global_summary_<timestamp>.csv
  summary/vfl_shap_<class>_summary_<timestamp>.csv
  outputs/vfl_shap_agent_mitigation_summary_<timestamp>.csv

Exact strings appear in the mitigation / summary cells of Train.ipynb.

================================================================================
DOWNSTREAM NOTEBOOKS (AFTER TRAINING / SHAP)
================================================================================

This repository does not stop at CSVs:

 Predict.ipynb  -> writes RAG_docs/predictions/predictions_*.json (per-flow SHAP + prediction summaries for inference samples)

  Plan.ipynb     -> RAG + LLM JSON action plans under RAG_docs/action_plans/

  Score.ipynb    -> NLP metrics on generated plans

So SHAP on the training notebook is for global / class-level analysis and
mitigation CSVs; per-sample agentic RAG uses the prediction JSON pipeline.
